1. 原仓库导读

一分钟结论

上游是 62 个术语的 7 层词典；本地变更有用，但要收敛展示层。

上游把 AI coding 的混乱归因于术语不透明，目标是用白话解释基础词，而不是写产品文档。

本文不做全文翻译，而是给读者三件事：原仓库结构导读、本地变更清单、以及这些变更是否有效的反思。

核心读法

  • model 不是 agent。model 负责生成；agent 是 model 加上下文、工具、权限和运行环境。
  • context 是预算。context window、attention budget、compaction 和 handoff 都在处理模型一次能看多少、能忠实处理多少。
  • tool 是边界。tool call、MCP、sandbox、permission 决定 agent 能否触碰文件、终端和外部系统。
  • 失败模式要转成动作。幻觉、迎合、知识截止和注意力衰减分别对应查证、反驳、加载新材料、清理或交接上下文。

2. 本地变更

这些变更的共同点是：借上游字典的结构启发，收紧 Harness Hub 自己的术语边界和发布边界，而不是复制上游词条。

  • docs/source-projects.md

    记录上游无许可证证据和 reference-only 决策，防止误把字典正文作为可复制素材。

    继续把 dictionary-of-ai-coding 放在 evaluated-but-not-installed 来源，而不是 capabilities install surface。

  • docs/harness-vocabulary.md

    把外部术语启发转成 Harness Hub 自己的概念边界，避免 model、agent、harness、skill、tool、check、review 混用。

    后续新增或修改本地术语时，优先更新这个文件并保持 local-original wording。

  • skills/effective-interact

    把词汇解释沉淀成 research-explainer 形态：先定义，再相邻概念边界，再使用场景、证据和验收边界。

    保留“不复制外部 glossary text”的硬边界，让 HTML 解释器只使用本地原创定义。

  • docs/skill-quality-guide.md

    把术语治理纳入技能质量规则，避免为了统一风格重写 imported skill bodies。

    术语调整只有在影响 routing、safety、validation 或 handoff meaning 时才改。

  • site/source-posts

    把这次中文译注和历史迭代记录变成 GitHub Pages 可发布的 source-backed public post。

    生成 post.json、source-ledger.json、effective-interact.input.json、index.html，并运行 source-post build/validate。

3. 反思结论

部分有效

effective-interact 的生成和验证链路有价值：它能把来源、判断、验证材料组织成可检查产物。但它不适合直接承担 blog 正文；上一版的问题正是公开页面太像交互报告，读者需要先理解组件和证据结构，才看得到结论。

冗余风险

继续扩成完整中文 glossary、CMS，或把正文写成 source-ledger 式审计页，都不值得。上游授权不明，本项目也没有高频维护双语词典的证据；扩写只会增加维护面和阅读成本。

下一步优化

保留术语边界和验证链路。公开文章只保留“原仓库导读、本地变更、反思结论”。除非上游补充授权，或本项目出现高频术语维护需求，否则不继续扩写。
