摘要

一分钟读完

先用正常文章语言回答：它是什么，为什么值得看，本项目为什么要管它。

结论：上游是 62 个术语的 7 层词典；本地变更有用，但要收敛展示层。

原仓库 `mattpocock/dictionary-of-ai-coding` 不是一个可安装工具，也不是一套工作流。它是一份 AI coding 术语词典：把初学者经常听到、但很难定位的词，按实际使用场景分组解释。

这篇文章只回答三个问题：

  • 原仓库到底讲了什么：62 个术语，分成 7 个层次。
  • 本项目根据它做了什么：补了本地术语边界、来源策略、解释器约束和发布记录。
  • 这些改动值不值：大部分有效，但展示层已经显得过重，应该收敛。

正文

核心思想

把词典翻成一个读者能带走的判断框架。

读者不用背词条，先抓住这几条线：

  • model 不是 agent：model 只生成文本；agent 是 model 加上上下文、工具、权限和运行环境。
  • context 是预算：上下文窗口、attention budget、compaction、handoff 都是在处理“模型一次能看多少、能忠实处理多少”。
  • tool 是边界：tool call、MCP、sandbox、permission 决定 agent 能不能碰文件、终端和外部系统。
  • 失败模式要转成动作：幻觉、迎合、知识截止、注意力衰减，分别对应查证、反驳、加载新文档、清理或交接上下文。
  • 自动化检查和人审不能混用：test/typecheck 是 deterministic check；agent review 是判断；human review 是人看真实 diff 或产物。

本地已做的变更

只列和 dictionary-of-ai-coding 直接相关的本地变化。

  • docs/source-projects.md：记录上游无许可证证据和 reference-only 决策，防止误把字典正文作为可复制素材。
  • docs/harness-vocabulary.md：把外部术语启发转成 Harness Hub 自己的概念边界，避免 model、agent、harness、skill、tool、check、review 混用。
  • skills/effective-interact：把词汇解释沉淀成 research-explainer 形态：先定义，再相邻概念边界，再使用场景、证据和验收边界。
  • docs/skill-quality-guide.md：把术语治理纳入技能质量规则，避免为了统一风格重写 imported skill bodies。
  • site/source-posts：把这次中文译注和历史迭代记录变成 GitHub Pages 可发布的 source-backed public post。

决策

反思结果

直接回答：有效吗、冗余吗、下一步怎么收敛。

对象结论冗余判断下一步
source-projects reference-only 记录有效。它把授权不明和用途边界写清楚，避免误复制上游正文。低。它是后续所有使用的约束入口。保留；若上游许可证变化，再重新评估能否扩大引用方式。

effective-interact 解释器和 fixture部分有效。它把术语解释变成可验证产物，但展示层容易过度。中高。上一版 blog 的问题就是交互组件多于读者问题。保留结构化生成和验证；公开文章优先导读、变更、反思。
skill-quality-guide 术语治理有效。它防止为了统一风格去改 imported skill body。低。它是维护政策，不是正文内容。继续作为审查规则；不要把它扩成第二套词典。
Pages source post当前版本需要收敛。发布链路有效，但文章结构必须服务读者。中。若每篇都生成 source-ledger 式报告，会不像 blog。固定本文三段式：原仓库导读、本地变更、有效性反思。

下一步优化

优化方向不是继续加内容，而是删掉读者不需要先看的复杂度。

现在该做：

  • 把文章重排为：原仓库内容树、本地变更、有效性与冗余反思。
  • 保留源码、source ledger 和 generated HTML 的验证链路。
  • 在最终交付中说明这不是未授权全文翻译。

继续观察：

  • 观察上游授权状态和术语更新。
  • 观察本项目后续是否真的需要更完整的中文 glossary；没有高频使用证据就不扩写。

暂时不做：

  • 不复制或逐条翻译上游 entry 正文。
  • 不把 dictionary-of-ai-coding 安装成 skill。
  • 不为了这篇文章引入 CMS、主题系统或新的 Pages 框架。
