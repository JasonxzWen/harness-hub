可信第三方评估为什么是 Harness 问题

状态：草稿

OpenAI 的可信第三方评估 playbook 不是一个可直接集成的 SDK，而是一套评估报告契约：说明评估主张、harness、预算、有效性风险和证据。Harness Hub 已有 validate-harness、agent-readiness 和 insight-* 基础，可以把它吸收为类似能力的报告准则。

Source Claims

降级 
- **fact** `claim-openai-role`: OpenAI frames trustworthy third-party evaluation as evidence for claims about frontier model capabilities and safeguards, not as a standalone benchmark score. Source: [A shared playbook for trustworthy third party evaluations](https://openai.com/index/trustworthy-third-party-evaluations-foundations/).
- **fact** `claim-openai-claim-types`: OpenAI separates evaluation claims into capability elicitation, safeguard performance, and comparison, and recommends reporting what claim the setup is meant to support. Source: [A shared playbook for trustworthy third party evaluations](https://openai.com/index/trustworthy-third-party-evaluations-foundations/).
- **fact** `claim-openai-harness`: OpenAI uses harness to mean the model-facing setup around a task, including prompts, tools, interfaces, control logic, memory, retries, validators, and related structures. Source: [A shared playbook for trustworthy third party evaluations](https://openai.com/index/trustworthy-third-party-evaluations-foundations/).
- **fact** `claim-openai-budget`: OpenAI argues that reports should expose budget details such as turns, tokens, retries, time, cost, and expected cost per successful solve where applicable. Source: [A shared playbook for trustworthy third party evaluations](https://openai.com/index/trustworthy-third-party-evaluations-foundations/).
- **fact** `claim-openai-validity`: OpenAI lists reward hacking, refusals, contamination, broken problems, and sandbagging as validity hazards that can distort evaluation results. Source: [A shared playbook for trustworthy third party evaluations](https://openai.com/index/trustworthy-third-party-evaluations-foundations/).
- **fact** `claim-openai-codex-floor`: For OpenAI capability evaluations, the article says evaluators are asked to use Codex as a common floor so tests run through an agentic interface users are likely to rely on. Source: [A shared playbook for trustworthy third party evaluations](https://openai.com/index/trustworthy-third-party-evaluations-foundations/).
- **fact** `claim-hh-role`: Harness Hub is a personal repo harness toolkit for making agent work repeatable across projects through analysis, skill installation, harness initialization, validation, and lifecycle commands. Source: [Harness Hub README](https://github.com/JasonxzWen/harness-hub/blob/0a815b5694e5663095fb3f59ebfc2a54da49637a/README.md).
- **fact** `claim-hh-validate`: Harness Hub's validate-harness surface already checks required files, QA boundaries, agent architecture boundaries, trigger hygiene, five subsystems, verification detection, and benchmark summary. Source: [Harness Hub README](https://github.com/JasonxzWen/harness-hub/blob/0a815b5694e5663095fb3f59ebfc2a54da49637a/README.md).
- **fact** `claim-hh-insight`: Harness Hub insight publishing keeps post.json, source-ledger.json, effective-interact input, and generated Pages HTML together with source attribution and fact/inference validation. Source: [Insight Publishing](https://github.com/JasonxzWen/harness-hub/blob/0a815b5694e5663095fb3f59ebfc2a54da49637a/docs/insight-publishing.md).
- **fact** `claim-hh-readiness`: Harness Hub agent-readiness analysis is read-only and reports context budget, outcomes, verification, routing, automation candidates, and learning capture instead of writing automation. Source: [Agent Readiness Analysis](https://github.com/JasonxzWen/harness-hub/blob/0a815b5694e5663095fb3f59ebfc2a54da49637a/docs/agent-readiness-analysis.md).
- **fact** `claim-hh-minimal-boundary`: Harness Hub's capability map keeps one minimal harness level and says new source ideas should improve that path rather than create new install tiers. Source: [Capability Map](https://github.com/JasonxzWen/harness-hub/blob/0a815b5694e5663095fb3f59ebfc2a54da49637a/docs/capability-map.md).
- **project-judgment** `claim-integration-shape`: For this repo, the practical integration is a reporting rubric for claim class, harness setup, resource budget, and validity checks, not a direct package dependency. Source: [A shared playbook for trustworthy third party evaluations](https://openai.com/index/trustworthy-third-party-evaluations-foundations/).
- **project-judgment** `claim-existing-primitives`: Harness Hub already has enough primitives to iterate toward evaluation-ready reporting: source-backed insight posts, harness validation, agent readiness, and Effective Interact reports. Source: [Harness Hub README](https://github.com/JasonxzWen/harness-hub/blob/0a815b5694e5663095fb3f59ebfc2a54da49637a/README.md).

Viewpoint Extraction

降级 
- `view-role`: 这个 OpenAI 项目的核心作用是给第三方评估报告定结构：先说明主张，再说明 harness、预算和有效性检查，最后让读者判断结论能泛化到哪里。 Trace: `claim-openai-role`, `claim-openai-claim-types`, `claim-openai-validity`.
- `view-harness-fit`: 它与 Harness Hub 的方向高度吻合，因为两边都把模型表现看成模型加运行环境的结果，而不是模型名本身的结果。 Trace: `claim-openai-harness`, `claim-hh-role`, `claim-hh-validate`.
- `view-integrate-as-rubric`: 可集成点应该是报告 schema 和验证门，而不是新增一个可安装 skill 套餐：给 validate-harness、agent-readiness 或 insight post 增加 claim、harness、budget、validity hazard 维度。 Trace: `claim-openai-budget`, `claim-integration-shape`, `claim-hh-minimal-boundary`.
- `view-current-iteration`: 本次博客本身就是一次 repo 迭代：它用 insight-* 把外部来源、项目判断和后续行动边界保存到 GitHub Pages，而不是只在会话里做口头总结。 Trace: `claim-hh-insight`, `claim-existing-primitives`.
- `view-not-enough-for-real-eval`: 如果要真的评测 agent 或模型，还缺任务集、评分器、预算、轨迹日志、人审策略和污染检查；当前最稳妥的下一步是文档化评估准备度，而不是声称已经能做第三方安全评估。 Trace: `claim-openai-validity`, `claim-hh-readiness`, `claim-integration-shape`.
- Assumption: 这里的“这个项目”指 OpenAI 文章提出的第三方评估方法论，而不是一个可安装代码仓库。
- Assumption: 这里的“发布到 GitHub Page”指生成并提交 site/ 下的 GitHub Pages 内容；实际部署仍需合并或推送到触发 Pages workflow 的分支。

影响

Project Mapping

降级 
- Integration: 把 OpenAI playbook 作为 Harness Hub 的 eval-report rubric 候选：claim type、tested system、harness、budget、elicitation、validity checks、evidence trail。
- Integration: 把该 rubric 优先映射到既有 surfaces：validate-harness 负责 repo harness 结构，agent-readiness 负责评估准备度，insight-* 负责公开来源账本和项目判断。
- Integration: 保持 minimal-only 架构：不新增 advanced/eval harness tier，不把第三方评估变成默认安装能力，先通过 source-backed blog 和后续 spec 收敛验收标准。
- **validate-harness**: 可以在未来增加 evaluation-report readiness 视角，检查目标 repo 是否能公开说明 harness、预算、scorer、日志和有效性风险。 Action: 若要实现，先写 SDD/spec 和测试；当前博客只记录参考价值，不改 CLI 行为。
- **agent-readiness**: 现有只读 categories 可扩展出 eval-readiness 解释层，判断 repo 是否具备可重复评测 agent 的上下文、结果、验证和学习沉淀。 Action: 把 OpenAI 的 claim/budget/validity 清单作为后续 report schema 候选。
- **insight-publishing**: 当前流程已经符合来源账本、事实/推断分离、Pages 输出和发布预检的需求。 Action: 用本篇博客验证 source-backed insight path，并把项目判断写入 post.json 与 source-ledger.json。
- **effective-interact**: 它可以作为评估报告的可读 HTML 层，把 claim、evidence、validation 和 next action 放在同一读者界面。 Action: 若后续做真实评估报告模板，再添加专门输入 fixture 和 validator。
- **workflow-router**: 第三方评估相关请求应先路由到一个 owner，再由 owner 选择 answer、review、hub-maintenance 或 SDD；避免 broad trigger 噪声。 Action: 本次仅手动记录路由低置信问题，不扩大 workflow-router 规则。
