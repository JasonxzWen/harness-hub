Supermemory 给 Harness Hub 的启发：AI 记忆不是更大的 RAG，而是可治理的上下文基础设施

这篇文章回答一个很具体的问题：Harness Hub 是否应该集成 Supermemory，或者把它作为迭代同类能力的参考？我的结论是：不进默认集成，但应该作为未来显式记忆能力的高价值参考。

来源：10 个主源 结论：reference-only 风险：外部凭证与自动捕获

结论：不默认集成，只作为高价值参考

Supermemory 是一个围绕 AI agent 的托管记忆与上下文层。它的工程价值很高，但它不是 Harness Hub 默认安装路径应该直接携带的东西。对这个仓库来说，正确做法是记录为 reference-only：吸收它的作用域、隐私、显式/隐式记忆分层和评估方法，而不是默认安装 hosted hook 或要求用户配置 API key。

Supermemory 到底是什么

如果只把它理解成“一个向量库”或“一个 RAG 服务”，就会错过它真正有价值的部分。Supermemory 在主仓库和文档里把自己描述为 AI 的 memory and context layer：它不是只保存文档块，而是把对话、文件、连接器数据和用户事实处理成可召回、可更新、可作用域隔离的上下文。

它的核心闭环很短：先用 profile 或 search 取出静态画像、动态近况和相关记忆，把这些上下文注入模型；对话结束后再把新的内容写回。这个闭环和 Harness Hub 关心的 agent harness 很接近，因为两者都在处理同一个问题：下一次 agent 开始工作时，怎样拿到足够准确、足够少、又不会越界的上下文。

Memory 记住用户和项目状态

用户偏好、项目决策、近期活动、过期或被新事实覆盖的信息。

RAG 检索静态知识

文档、代码、网页和连接器同步来的资料，适合回答“资料里有什么”。

Profile 压缩成可注入上下文

把长期事实和短期动态拆开，避免每次都塞完整历史。

为什么不直接集成进 Harness Hub 默认路径

默认集成的标准不是“有没有用”，而是“是否符合 Harness Hub 的最小、显式、可回滚、可验证边界”。Supermemory 在产品上有价值，但默认接入会引入四类新责任。

责任 具体问题 对 Harness Hub 的影响
凭证 需要 Supermemory API key 或 OAuth。 默认 harness 不应该把外部凭证变成安装前提。
远程写入 自动 capture 会把对话或项目知识发到托管服务。 这必须是明确选择，而不是安装技能后的隐式副作用。
作用域 用户记忆、项目记忆、团队记忆需要严格隔离。 一旦作用域错配，记忆污染比普通检索错误更难排查。
部署边界 官方 self-hosting 文档是企业部署路径，需要部署包和外部基础设施。 不能把主仓库误读成“可本地直接托管的记忆引擎”。
项目判断：默认分发里放一个会自动远程写记忆的 hook，和 Harness Hub 当前的 lifecycle policy 冲突。它可以成为 opt-in 集成，但不能变成 minimal harness 的一部分。

真正值得借鉴的不是代码，而是记忆治理结构

Supermemory 生态里已经有针对不同 agent host 的迭代：codex-supermemory、opencode-supermemory、claude-supermemory、smfs 和 MemoryBench。这些项目比主仓库更能说明它对 Harness Hub 的启发。

Codex plugin 隐式 hooks + 显式 fallback skills

自动召回和自动捕获解决连续性；手动 search/save/forget/login 给用户保留控制权。

OpenCode plugin 项目记忆、代码库索引、压缩前保存

把 compaction 当成记忆边界事件，而不是等上下文丢失后再补救。

smfs 把记忆暴露成文件系统

agent 已经会读写文件和 grep；把记忆做成文件接口，学习成本比新工具低。

MemoryBench 把记忆质量变成可跑的评估

ingest、index、search、answer、evaluate、report，比“看起来召回到了”更接近工程验收。

这些模式可以转成 Harness Hub 的未来约束：如果要做 memory provider，必须先定义用户/项目作用域、private redaction、dry-run、卸载、状态检查和 benchmark fixture。否则“持久记忆”会从能力变成不可解释的上下文污染源。

如果 Harness Hub 以后要做类似能力，应该从哪里开始

不是从安装 Supermemory hook 开始，而是从一个本地、可读、可验证的 context ledger 开始。这个 ledger 可以先记录来源、结论、作用域、过期条件和证据，再决定是否接入远程 provider。

  1. 第一步：写设计，不写 hook。定义哪些内容能成为记忆、谁能读、多久过期、如何撤销、如何审计。
  2. 第二步：做本地 provider。先用本地 JSONL 或 SQLite 验证作用域、召回和遗忘语义，避免凭证和远程依赖干扰。
  3. 第三步：加 benchmark fixture。用几组真实 repo-harness 场景测试召回是否帮忙，而不是只测试 API 成功。
  4. 第四步：再做 opt-in remote adapter。Supermemory 可以是候选 provider，但安装必须显式、可 dry-run、可 status、可 uninstall。
