# Report Shape

Default to a private Markdown report for the analysis record. For high-volume, multi-session, multi-case, or option-heavy audits, also generate an `effective-interact` visual-report input and validate the resulting HTML artifact when filtering, navigation, or comparison across many evidence items lowers interpretation cost.

## Required Order

1. `BLUF`
2. `Evidence Coverage`
3. `Top Bottlenecks`
4. `Top Recommendations`
5. `Task Profile`
6. `Trace Audit`
7. `Core Positioning`
8. `Unknowns`
9. `Evidence Appendix`
10. `Warnings`

## Evidence IDs

Every strong bottleneck or recommendation must cite event IDs from the ledger. Use candidate IDs only as weak support.

Bottleneck and recommendation evidence should prefer primary interaction events: confirmed, exact-or-strong repo affinity, non-low confidence, and interaction role. Use repo-state evidence for background context, not as the direct cause of a top recommendation.

## Task And Trace Detail

- Task profile should include clustered interaction traces, not only global counts.
- Trace audit should distinguish parsed tool-call/tool-result events, failed or blocker-linked tool events, repeated failed tools, and environment-linked failures.
- Successful handoffs and state updates can appear in task clusters, but should not become Top Bottlenecks without explicit friction evidence.

## Language

Match the user's language in the final answer. The report itself may be English if generated by the script, but the agent-facing answer should explain the result in the user's language.

## Avoid

- long chronology without judgment;
- generic productivity advice;
- public-post framing;
- private evidence copied into tracked docs;
- unsupported statements about user preference or intent.
