# HTML Aesthetic Preflight

Source: selective review of `Leonxlnx/taste-skill` at `3c7017d636c3a4aad378433ea6d0cfa6c921da4a` (MIT).

Use this reference only for `effective-interact` HTML reports and interaction artifacts. Do not apply marketing-page rules to production UI work from this report validator. Production UI still routes to `frontend-design`, anti-template landing/portfolio/redesign direction routes to `design-taste-frontend`, and complex standalone artifacts still route to `web-artifacts-builder`.

## Report-Safe Adaptation

The useful transfer is the pre-flight discipline, not the whole frontend framework:

- Reject default AI-purple and blue/purple glow unless the report subject, source evidence, or brand tokens explicitly require it.
- Avoid "default card grid" output. Cards must clarify hierarchy, status, evidence, or options; otherwise use spacing, tables, source rails, diagrams, or grouped navigation.
- Keep layout families varied in long artifacts. A report with many sections should mix summary cards, tables, timelines, diagrams, code, evidence, and actions instead of repeating one card stack.
- Keep hero areas compact. The hero answers the reader's first question; it is not a landing-page billboard.
- Keep section headers vertically honest. Avoid decorative split headers whose right-hand paragraph repeats the heading.
- Use one visual system per artifact: one neutral base, one accent family, one radius rule, and status colors with meaning.
- Treat typography and spacing as scanning tools: short paragraphs, compact cards, stable table columns, clear code line height, no viewport-scaled font tricks.
- Motion must explain state, focus, or navigation. It must have `prefers-reduced-motion` coverage and must not be decorative spectacle.
- Run a pre-flight before handoff: palette, card purpose, section repetition, hero density, contrast, reduced motion, mobile collapse, overflow, and copy clarity.

## Boundary

This preflight is intentionally stricter than ordinary report writing and narrower than frontend design. It should improve generated HTML reports without changing how Harness Hub routes normal UI creation, code explanation, Chinese answers, or plain Markdown handoffs.
